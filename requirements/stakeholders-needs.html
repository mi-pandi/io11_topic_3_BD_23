# Запити зацікавлених осіб

# Вступ

Даний розділ містить опис роботи, запитів зацікавлених осіб, функціональності системи аналізу вмісту тексту та перевірки на плагіат.

# Мета

Ціль проєкту - розроблення надійної системи аналізу медіаконтенту (тексту) з наданням простого та зручного інтерфейсу користувача. Інтерфейс має забезпечувати навігацію між аналізом тексту та перевіркою вмісту на плагіат.

# Контекст

Документ впроваджує технічне завдання системи, вимоги надійності та функціоналу, взаємодію між користувачем, системою та адміністратором.

# Характеристика ділових процесів

Учасниками процесу є користувач, система та адміністратор. У цьому розділі наведені сценарії взаємодії користувача, системи та адміністратора, які можуть виникати при роботі.

# Користувач

1. ID: USER.REGISTRATION
НАЗВА: Авторизувати користувача і створити йому обліковий запис
УЧАСНИКИ Користувач, система
ПЕРЕДУМОВИ: Користувач повинен мати бажання зареєструватися і користуватися системою
РЕЗУЛЬТАТ: Користувач отримує власний обліковий запис, отримує доступ до облікового запису
ВИКЛЮЧНІ СИТУАЦІЇ: Користувач не має бажання реєструватися
ОСНОВНИЙ СЦЕНАРІЙ: 1. Користувач починає взаємодію
2. Користувач натискає на кнопку входу для реєстрації на сайті
3. Система надсилає користувачу форму для вводу даних для створення облікового запису
4. Користувач вводить дані свого облікового запису у форму входу реєстрації
5. Користувач натискає на кнопку посилання системі даних для реєстрації
6. Система отримує дані, які надіслав користувач
7. Система підтверджує дані, які надіслав користувач
8. Система створює запис про додавання користувача
9. Система додає користувача до системи
10. Користувач отримує доступ до свого облікового запису
11. Користувач завершує взаємодію.
2. ID: USER.LOGIN
НАЗВА: Авторизувати користувача в обліковий запис в систему для того, щоб він міг нею користуватися
УЧАСНИКИ: Користувач, система
ПЕРЕДУМОВИ: Користувач має бути зареєстрований в системі
РЕЗУЛЬТАТ: Користувач отримує доступ до облікового запису
ВИКЛЮЧНІ СИТУАЦІЇ: Користувач не є зареєстрованим на сайті
ОСНОВНИЙ СЦЕНАРІЙ: 1. Користувач починає взаємодію.
2. Користувач натискає на кнопку входу в наявний обліковий запис
3. Система надсилає користувачу форму для вводу даних наявного облікового запису
4. Користувач вводить дані свого облікового запису у форму входу в систему
5. Користувач натискає на кнопку посилання системі даних наявного облікового запису
6. Система отримує дані, які надіслав користувач
7. Система підтверджує даних, які надіслав користувач
8. Система надає користувачу доступ до його облікового запису
9. Користувач отримує доступ до свого облікового запису
10. Користувач завершує взаємодію.
3. ID: USER.GET_DATA
НАЗВА: Отримання користувачем даних
УЧАСНИКИ: Користувач, система
ПЕРЕДУМОВИ: Користувач має бути зареєстрований на сайті та бути у своєму обліковому записі
РЕЗУЛЬТАТ: Користувач отримує дані про свій текст і коефіцієнт плагіату цього тексту з другим текстом або вже наявним текстам в базі даних
ВИКЛЮЧНІ СИТУАЦІЇ: Користувач не є зареєстрованим на сайті, у форми для заповнення нічого заданого
ОСНОВНИЙ СЦЕНАРІЙ: 1. Користувач починає взаємодію
2. Користувач записує в форми для текстів дані для порівняння
3. Він натискає кнопку заповнення форми для початку порівняння текстів
4. Користувач за бажанням може вибирає функцію порівняння свого тексту з наявним текстом в базі даних
5. Система отримує два тексти
6. Система починає обробку текстів
7. Якщо вказано функцію порівняння з наявним, то система починає порівнювати тексти один з одним
8. Система генерує вихідні дані за запитом користувача
9. Система передає вихідні дані користувачу про порівняння
10. Якщо було вказано зробити порівняння вже з наявними текстами, то система також виводить ці дані
11. Користувач отримує розгорнуті дані про порівняння двох текстів та інші параметри цих текстів
12. Користувач завершує взаємодію
4. ID: USER.SAVE_TEXT
НАЗВА: Збереження тексту до бази даних
УЧАСНИКИ: Користувач, система
ПЕРЕДУМОВИ: Користувач має бути зареєстрований на сайті та бути у своєму обліковому записі
РЕЗУЛЬТАТ: Користувач зберігає текст до бази даних системи
ВИКЛЮЧНІ СИТУАЦІЇ: Користувач не є зареєстрованим на сайті, користувач не хоче зберігати свій текст
ОСНОВНИЙ СЦЕНАРІЙ: 1. Користувач починає взаємодію
2. Виконується процес USER.GET_DATA
3. Він натискає кнопку заповнення форми для зберігання текстів до бази даних
4. Система зберігає текст до своєї бази даних
5. Користувач завершує взаємодію
5. ID: USER.HELP
НАЗВА: Звернутись до системи за допомогою
УЧАСНИКИ: Користувач, cистема
ПЕРЕДУМОВИ: Користувач виконав авторизацію в системі та потребує інструкцій
РЕЗУЛЬТАТ: Довідка про роботу з системою
ВИКЛЮЧНІ СИТУАЦІЇ: Система не змогла ідентифікувати запит користувача(код помилки: user.help_err1)
ОСНОВНИЙ СЦЕНАРІЙ: 1. Користувач починає взаємодію
2. Користувач натискає на кнопку звернення до системи
3. Система надає користувачу форму для заповнення
4. Користувач отримує доступ для заповнення форми
5. Користувач вписує текст свого питання
6. Користувач надсилає заповнену форму системі
7. Система аналізує проблему користувача за ключовими словами
8. Система надсилає користувачу необхідну довідку
9. Користувач отримує довідку та ознайомлюється з нею
10. Користувач завершує взаємодію.

# Короткий огляд продукту

Система для аналізу текстів, використовуючи різні параметри, порівняння текстів між собою на предмет плагіату першого тексту відносно другого.

# Функціональність

У цьому сервісі є два учасники, які мають різні можливості та функції. Нижче перелічені можливості цих учасників.

Користувач має такі можливості:

  • Має власний обліковий запис для користування системою, надсилати запит для аналізу та порівняння.
  • Порівнювати тексти з вже наявними та зберегти свій текст до бази даних.
  • Дати системі на аналіз текст та окремо виводити його параметри: мова використана у тексті, кількість символів, слів, речень, та інші.
  • Отримати діаграму, що містить інформацію скільки разів у тексті зустрічаються деякі окремі слова.
  • Отримати інструкцію до використання.

Адміністратор має такі можливості:

  • Отримує та опрацьовує запит користувача, керує його правами.
  • Додає нових користувачів із правами адміністратора.
  • Додає нові тексти до бази диних, змінює їх зміст і параметри за необхідності, а також змінює інші параметри в базі даних.
  • Сортує тексти за різними параметрами.
  • Створює групи користувачів за крайної необхідності.

# Практичність

Із практичністю ми маємо на увазі здатність програми відповідати вимогам користувача, її зручнісь у використанні, та ефективність у покритті потреб користувача.

  • Наявність user-friendly інтерфейсу, яким буде зручно користуватися.
  • Можливість завантажити оброблений текст до бази даних для покращення роботи проекту і порівняння із вже існуючими текстами у базі даних, завантаженими іншими користувачами.
  • Отримана інформація буде представлена наочно та зрозуміло для користувача.
  • Наявність зворотнього зв'язку для звернення до розробників.
  • Наявність інструкції до використання на випадок виникнення труднощів у користувача під час роботи із інтерфейсом.

# Надійність

Надійність - це здатність продукту тримати рівень працездатності при навантаженнях і у "бойовму" режимі роботи сервера.

  • Обробка неправильних запитів, система буде про них сповіщати користувача.
  • Авторизація до системи відбуватиметься по токенах, які прив'язуються до окремого користувача системи лише на один сеанс і видалятимуться при виході користувача з системи, а при повторному вході виділятимуться нові. Така схема запобігає взлому акаунтів за одної наявності паролю і логіну користувача. Також ромір такого токену складає більше ніж 16 символів, що ускладнює процес взлому акаунту та несанкціоноване отримання даних користувача.

# Продуктивність

Під продуктивністю ми розуміємо здатність системи правильно використовувати ресурси для виконання поставлених завдань за невеликий проміжок часу.

  • Можна зберігати свої тексти вже с існуючими текстами в системі, для подальшого використання або, наприклад, для того щоб ним могли користуватися ініші користувачі.
  • Здатність системи швидко виводити вихідні дані.
  • Обмеження розміру тексту для вводу складає 500 тисяч символів.

# Експлуатаційна придатність

Есплутаційна придатність - це властивість проекту, яка дозволяє підтримувати проект і виправляти помилки або вдосконалювати продукт за необхідності.

  • Для користування веб-додатком можна користуватися браузером з будь-якого девайса.
  • При необхідності до системи можна додати нові модулі обробки тексту які будуть видавати користувачу новий функціонал системи.
  • За наявності багів, вони будуть виправлятися розробниками.
  • Буде наявна документація до проекту для нових розробників у команді.
  • Код Back-End частини буде написаний на мові Python - код на цій мові програмування є легкопідтримуваним.
